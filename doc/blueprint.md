# Blueprint

## Назначение документа

`ts_connect` уже опирается на Metarhia/Impress как на базовый runtime, контрактный слой RPC и модель исполнения процедур. Поэтому его нужно сопровождать как Metarhia/Impress сервис с интеграцией к TradeStation, а не как набор независимых ad hoc RPC-методов. Этот документ фиксирует, на какие уровни и инварианты должен опираться проект: runtime-контракт Impress, разделение слоев `api/domain/lib/types`, lifecycle stream-подписок и правила безопасного развития интеграции с TradeStation.

Документ опирается на фактическое поведение Impress:

- `parameters` и `returns` валидируются через `Schema.from(...)` в `impress/lib/procedure.js` и `impress/lib/bus.js`;
- `validate` вызывается после проверки `parameters` и до вызова `method`;
- `DomainError` конвертируется в transport error через словарь `errors`;
- `.d.ts` рядом с API-процедурой является compile-time контрактом и не добавляет runtime-поведения сам по себе.

## На что должен опираться проект

Проект должен опираться одновременно на четыре основания:

- Metarhia/Impress как runtime RPC-процедур, валидации и исполнения контрактов;
- `application/domain` как слой server-side состояния и lifecycle-логики;
- `application/lib` как слой интеграции с TradeStation и технических adapter/parser helper-ов;
- документацию и review-checklist как способ удерживать совместимость и не размывать архитектурные границы.

Проект не должен опираться на случайные локальные решения уровня одного endpoint. Если важное правило живет только в отдельном `application/api/*.js` и не выражено через контракт, domain-layer или документацию, это архитектурный долг.

## Слои сервиса

### `application/api`

Публичный RPC-контракт сервиса. Каждый файл в `application/api/*` должен описывать одну Metarhia-процедуру и быть минимальным слоем orchestration:

- принимать и нормализовать параметры;
- вызывать `domain.*` и `lib.*`;
- возвращать структурированный результат;
- формировать доменные ошибки через `DomainError`, если ошибка является ожидаемой частью публичного контракта.

API-слой не должен:

- хранить состояние stream-ов напрямую;
- дублировать низкоуровневую reconnect-логику;
- разбирать транспортные детали ответа TradeStation глубже, чем требуется для публичного контракта.

### `application/domain`

Слой серверного состояния и orchestration, который живет дольше одного RPC-вызова. Здесь размещаются:

- клиентские registry;
- multiplex stream-подписок;
- shared state между несколькими Metacom-клиентами;
- операции cleanup по `client.close`, idle timeout и другим lifecycle-событиям.

Если поведение связано с жизненным циклом подписки, reuse между несколькими клиентами или server-side state, оно должно быть в `domain`, а не в `api`.

### `application/lib`

Интеграция с внешним миром и чистые технические helper-ы:

- TradeStation HTTP/stream adapters;
- parser-ы и reader-ы ответов;
- утилиты нормализации символов, endpoint и query params;
- server-side stream helper-ы, которые публикуют данные, но не принимают архитектурные решения за `domain`.

`lib` не должен брать на себя политику публичного API. Его задача: выполнить техническую работу и вернуть предсказуемый результат.

### `types`

Глобальная и сервисная типизация. Сейчас в проекте есть только `types/global.d.ts`, которая объявляет глобальные namespace `api`, `lib`, `domain`, а также `DomainError`.

Локальные `.d.ts` рядом с `application/api/*` пока не внедрены как массовый паттерн. Для этого сервиса это целевой стандарт, а не уже завершенная миграция.

## Обязательный контракт API-процедуры

### Именование и плотность кода

Локальные переменные, helper-ы и временные значения должны называться коротко, но полно:

- только `camelCase`;
- 1-4 слова;
- без сокращений, если можно использовать полное слово;
- без учета веса или центральности сущности;
- код писать плотно, без декоративных пустот, ближе к стилю Impress и существующего кода проекта.

### Что должно быть в `.js`

Новая публичная процедура должна стремиться к следующему набору полей:

- `access` для явного уровня доступа;
- `parameters` для schema validation входа;
- `returns` для schema validation результата;
- `errors` для словаря доменных кодов ошибок;
- `validate`, если нужно дополнительное правило поверх схемы;
- `method` как единственная точка выполнения бизнес-операции.

Минимальный runtime-паттерн:

1. `parameters` проверяет форму входа.
2. `validate` проверяет бизнес-ограничения, которые не выражаются схемой.
3. `method` вызывает `domain` и `lib`.
4. `returns` проверяет shape результата.
5. `DomainError` используется только для ожидаемых бизнес-ошибок, перечисленных в `errors`.

### Когда возвращать `DomainError`

`DomainError` нужно использовать для предсказуемых нарушений публичного контракта:

- недопустимое бизнес-значение параметра;
- неподдерживаемый режим, который сервис осознанно объявляет как ограничение;
- конфликт состояния, который клиент должен обработать штатно.

`throw Error(...)` нужен для внутренних дефектов, unexpected transport-проблем, программных ошибок и случаев, которые не должны становиться частью стабильного API-контракта.

Практическое правило:

- если код ошибки должен жить в `errors` и быть понятен клиенту, это `DomainError`;
- если ошибка означает баг, интеграционный сбой или поломку инварианта, это исключение.
- если метод исторически принимает несколько legacy-представлений одного входа, сначала нужно выполнить совместимую нормализацию, а уже потом применять domain-ограничения.
- если внешний параметр нельзя корректно отобразить в поддержанный upstream format, нужно вернуть domain error, а не молча падать в дефолтный режим.
- если одинаковая нормализация action-like параметров повторяется в нескольких endpoint-ах, ее нужно выносить в общий helper уровня `lib.utils.normalizeAction`.

### Когда использовать `validate`

`validate` нужен, когда проверка важна для доменной логики, но неудобна в виде metaschema-описания. Примеры:

- зависимость одного параметра от другого;
- запрет комбинации `stream=false` и `action=touch`;
- ограничения, завязанные на внешнее состояние.

Нормализация значений, не меняющая смысл контракта, допустима внутри `method`, но должна быть явной и предсказуемой.

## Что дает `.d.ts` рядом с API-процедурой

Эталонный пример Metarhia-стиля:

- `application/api/example.1/add.js`
- `application/api/example.1/add.d.ts`

Смысл `.d.ts` в таком паттерне:

- объявить namespace контракта API для TypeScript;
- сузить допустимые коды ошибок, например `'EARGA' | 'EARGB'`;
- дать IDE и статическому анализу явный `CustomError extends DomainError`;
- зафиксировать compile-time документацию рядом с runtime-процедурой.

Что `.d.ts` не делает:

- не валидирует runtime-параметры;
- не заменяет `parameters`, `returns` и `errors` в `.js`;
- не генерируется автоматически текущим сервисом;
- не влияет на `impress/lib/procedure.js`, пока не используется TypeScript-инструментами.

Для `ts_connect` правило такое:

- `.d.ts` рекомендуется для публичных процедур с доменными кодами ошибок или нетривиальным namespace-контрактом;
- отсутствие `.d.ts` должно быть осознанным, а не случайным;
- при добавлении `.d.ts` имена кодов ошибок должны совпадать с `errors` в `.js`.

Первый закрепленный пример внутри этого репозитория: `application/api/options/chain.js` и `application/api/options/chain.d.ts`. Для этого метода `.d.ts` считается reference pattern для публичной процедуры с доменными кодами ошибок.
Второй reference pattern для stream-метода: `application/api/stream/addBarchart.js` и `application/api/stream/addBarchart.d.ts`.

## Правила для stream API

Stream-методы сервиса должны выглядеть как управляемый lifecycle, а не как одноразовый старт потока.

### Базовый lifecycle

- `subscribe` создает или переиспользует upstream stream и регистрирует downstream-подписчика;
- `touch` продлевает жизнь подписки;
- `unsubscribe` снимает подписчика и при необходимости останавливает upstream stream;
- `streamKey` должен быть стабильным идентификатором подписки;
- idle cleanup должен работать без участия клиента после истечения TTL;
- `client.close` должен очищать подписки автоматически.

### Разделение ответственности

- `application/lib/ts/stream.js` отвечает за lifecycle одного upstream HTTP stream к TradeStation;
- `application/domain/ts/streams.js` отвечает за multiplex подписчиков и cleanup;
- `application/api/stream/*` и `application/api/options/chain.js` должны вызывать managed subscription layer, а не управлять registry вручную.

### Диагностика

Каждый значимый stream-сценарий должен иметь способ проверить:

- какие upstream stream-ы активны;
- какие downstream-подписки живут на сервере;
- по какому `streamKey` они адресуются;
- очищаются ли они после `unsubscribe`, `client.close` и idle timeout.

Для этого допустимы диагностические API-методы уровня `application/api/info/client.js`.

## Правила обновления сервиса

### Политика размера task и закрытия работы

- если пользователь не ограничил задачу явно одним узким изменением, task нужно собирать крупно и закрывать совместимый набор связанных доработок за один проход;
- после выполнения любого task обязателен отдельный review-этап;
- после review нужно обновить все файлы в `doc/`, а не только один затронутый документ;
- task считается закрытым только в последовательности `implement -> review -> update doc/*`.

### Правила ведения doc/*

- `blueprint.md` — текущее состояние продукта, целевой контракт, обязательные правила. Историю и закрытые фазы не хранит — для этого есть git history.
- `task.md` — рабочий журнал задач блоками; архивируется при накоплении >30 завершённых задач или закрытии крупного цикла.
- `review.md` — постоянный checklist + заключения по блокам; архивируется вместе с task.md.
- `changelog.md` — архив закрытых задач и review-заключений; загружается вручную.

### Добавление нового API-метода

1. Сначала определить публичный контракт: вход, выход, доменные ошибки, ограничения.
2. Зафиксировать, где должна жить логика: `api`, `domain` или `lib`.
3. Описать `parameters`, `returns`, `errors`.
4. При необходимости добавить `.d.ts` как compile-time контракт.
5. Проверить, не ломает ли изменение уже существующие клиенты.
6. Smoke-проверка: happy path, минимум одна доменная ошибка, минимум один invalid input, shape результата соответствует `returns`.

### Изменение stream endpoint

1. Сначала определить upstream endpoint TradeStation и downstream event contract.
2. Не хранить stream state в `application/api/*`.
3. Выделить stable `streamKey`.
4. Продумать cleanup при `unsubscribe`, timeout и `client.close`.
5. Добавить диагностический сценарий и smoke-проверку.
6. Smoke-проверка: `subscribe` → `streamKey` → events → `touch` → `unsubscribe` → cleanup verified.

### Совместимость

Публичный RPC-метод считается совместимым, если:

- не удалены существующие обязательные поля без миграции;
- старые параметры продолжают работать или явно помечены deprecated;
- коды доменных ошибок не изменены без синхронного обновления клиентов;
- ограничения и неподдержанные режимы задокументированы явно.

### Ограничения должны фиксироваться в документации

Если сервис чего-то не поддерживает, это нужно фиксировать явно. Для текущего состояния `ts_connect` это особенно важно для:

- option chain режимов beyond `spreadType = 'Single'`;
- частично реализованных market data stream endpoint-ов;
- методов, где ответ TradeStation требует defensive guards.

Ограничение должно быть объявлено как часть сервиса, а не оставаться неявным знанием в коде.

### Порядок работы над изменением

**Перед изменением:**

- сверить текущий код с реальным поведением Impress и TradeStation;
- проверить, не дублирует ли новый код уже существующий helper или stream manager;
- определить совместимость с текущими клиентами Metacom.

**Во время изменения:**

- не переносить domain state в `application/api/*`;
- не расширять публичный контракт без явного описания `parameters`, `returns`, `errors`;
- не добавлять неподдержанный режим без явного ограничения или ошибки;
- не скрывать технический долг под универсальными `try/catch`.

**После изменения:**

- проверить типы, линт и локальные smoke-сценарии;
- провести отдельный review по постоянному checklist;
- обновить все `doc/*`, если появилось новое устойчивое правило сервиса или новая категория риска;
- зафиксировать ограничения, которые должны знать сопровождающие.
