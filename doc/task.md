# Task

## Назначение документа

Этот документ описывает порядок безопасного обновления `ts_connect`. Он не фиксирует разовую историю изменений, а задает рабочий чеклист для инженера, который добавляет новый RPC-метод, stream endpoint или меняет существующую интеграцию с TradeStation.

Перед любым изменением нужно помнить: проект уже опирается на Metarhia/Impress runtime и должен развиваться внутри этой модели. Практически это означает, что любое изменение сначала проверяется относительно contract layer (`access`, `parameters`, `returns`, `errors`), затем относительно слоев `api/domain/lib`, и только потом реализуется как код конкретного endpoint.

## На что нужно опираться при изменении сервиса

- на фактическое поведение Impress, а не на предположение, как "должен" работать RPC;
- на разделение ответственности между `application/api`, `application/domain` и `application/lib`;
- на уже существующие lifecycle-паттерны stream-ов и cleanup;
- на совместимость публичного контракта для клиентов Metacom;
- на явно задокументированные ограничения TradeStation-интеграции.

## Базовый порядок работы

1. Изучить текущий контракт метода и фактическое поведение TradeStation endpoint.
2. Определить, где должна жить логика: `application/api`, `application/domain` или `application/lib`.
3. Описать или скорректировать `parameters`, `returns`, `errors`.
4. Проверить, нужны ли `validate` и `.d.ts`.
5. Реализовать изменение без переноса server-side state в API-слой.
6. Провести smoke-проверку и review по постоянному checklist.
7. Обновить все файлы `doc/*` по ограничениям, совместимости и новым правилам после review.

## Политика размера task

- если пользователь не задал узкую границу явно, task нужно делать большим и закрывать максимум совместимых доработок за один проход;
- после каждого task обязателен отдельный review;
- после review обязателен проход по всем файлам `doc/*`, даже если правило меняется только в одном месте;
- только после этой последовательности task считается завершенным.

## Чеклист для нового API-метода

### Контракт

- определить `access`;
- описать `parameters` через metaschema-совместимый объект;
- описать `returns`, если метод возвращает структурированные данные;
- перечислить ожидаемые доменные ошибки в `errors`.
- привести новые локальные переменные к коротким полным именам в `camelCase`, 1-4 слова, без учета веса центральных сущностей, и держать код плотным в стиле Impress.

### Валидация и ошибки

- использовать `DomainError` только для ожидаемых ошибок публичного контракта;
- не заменять внутренние дефекты и transport failures доменными ошибками;
- использовать `validate`, если правило не выражается схемой;
- не оставлять бизнес-ограничения только в тексте ошибки без явной проверки.
- при контрактизации legacy-метода сначала нормализовать старые falsey/string представления входа, чтобы не сломать совместимость случайной строгой проверкой.
- если значение нельзя без искажения перевести в поддержанный upstream interval или mode, вернуть domain error вместо тихого fallback.
- если один и тот же способ нормализации action-like полей встречается в нескольких методах, вынести его в общий helper, а не копировать локально.

### Типизация

- проверить, нужен ли `.d.ts` рядом с процедурой;
- если у метода есть доменные коды ошибок или сложный namespace-контракт, предпочесть `.d.ts`;
- синхронизировать коды ошибок между `.js` и `.d.ts`;
- не считать `.d.ts` заменой runtime validation.

Текущее правило для репозитория:

- `application/api/options/chain.js` и `application/api/options/chain.d.ts` использовать как reference example для публичного метода с явным `errors` и локальным типизированным `CustomError`.
- `application/api/stream/addBarchart.js` и `application/api/stream/addBarchart.d.ts` использовать как reference example для stream-метода с явным `errors`, lifecycle actions и локальным типизированным `CustomError`.

### Smoke-проверка

- проверить happy path;
- проверить минимум одну доменную ошибку;
- проверить минимум один invalid input case;
- убедиться, что shape результата соответствует `returns`.

## Чеклист для изменения stream-логики

### Архитектурное размещение

- upstream stream lifecycle должен жить в `lib` и `domain`, а не в `api`;
- shared subscription registry должен находиться в `domain`;
- API-метод должен вызывать managed subscription layer, а не напрямую манипулировать внутренними registry.

### Lifecycle

- определить `subscribe`, `touch`, `unsubscribe`;
- ввести стабильный `streamKey`;
- предусмотреть cleanup при idle timeout;
- предусмотреть cleanup при `client.close`;
- исключить reconnect после осознанного ручного `stop`.

### Диагностика

- обеспечить способ увидеть активные upstream stream-ы;
- обеспечить способ увидеть активные downstream subscriptions;
- иметь сценарий ручной проверки cleanup;
- при необходимости обновить диагностические методы уровня `application/api/info/client.js`.

### Smoke-проверка stream endpoint

1. Вызвать `subscribe`.
2. Получить или сверить `streamKey`.
3. Убедиться, что приходят события нужного типа.
4. Вызвать `touch`, если поток должен жить дольше default TTL.
5. Вызвать `unsubscribe`.
6. Проверить, что подписка и upstream stream очищены.

## Порядок обновления всего сервиса

### Перед изменением

- сверить текущий код с реальным поведением Impress и TradeStation;
- проверить, не дублирует ли новый код уже существующий helper или stream manager;
- определить совместимость с текущими клиентами Metacom.

### Во время изменения

- не переносить domain state в `application/api/*`;
- не расширять публичный контракт без явного описания `parameters`, `returns`, `errors`;
- не добавлять неподдержанный режим без явного ограничения или ошибки;
- не скрывать технический долг под универсальными `try/catch`.

### После изменения

- проверить типы, линт и локальные smoke-сценарии;
- провести отдельный review по постоянному checklist;
- обновить все `doc/*`, если появилось новое устойчивое правило сервиса или новая категория риска;
- зафиксировать ограничения, которые должны знать сопровождающие.

## Специфика `ts_connect`, которую нужно помнить всегда

- не все методы `application/api/*` еще приведены к полному Metarhia-стандарту;
- локальные `.d.ts` рядом с API-процедурами пока не внедрены в проект массово;
- TradeStation stream-методы требуют явного lifecycle и cleanup;
- option chain безопасно поддержан только в режиме `spreadType = 'Single'`, пока не спроектирован отдельный контракт для других стратегий;
- ответы TradeStation в account/order частях сервиса нужно читать defensive-образом, не доверяя shape без проверок.

## Пример применения стандарта к текущему сервису

Для `api.options.chain` изменения считаются корректными, если одновременно выполнены условия:

- snapshot и stream режимы разделены явно;
- `subscribe` / `touch` / `unsubscribe` проходят через managed stream layer;
- `streamKey` стабилен и пригоден для повторного обращения;
- неподдержанные option spread types не маскируются под рабочий режим;
- диагностический метод позволяет увидеть upstream и subscriptions отдельно.
